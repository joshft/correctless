# Spec: Add /cexplain Skill for Guided Codebase Exploration

## What

Add a `/cexplain` skill that provides guided, interactive codebase exploration using mermaid diagrams and prose walkthroughs. Instead of generating a static report with every diagram type, the skill starts with a 30-second project overview and lets the user drill into what they care about. Each answer offers natural follow-ups. The user can export what they explored to an HTML document at any point. Serena MCP dramatically improves trace quality but the skill works without it. Both Lite and Full get it.

## Rules

- **R-001** [integration]: When invoked, `/cexplain` scans the project and presents a 30-second overview: project name, language(s), approximate LOC, one-sentence purpose, and a numbered list of major components (3-7) with their locations and one-line descriptions. The overview is derived from reading AGENT_CONTEXT.md (if it exists), ARCHITECTURE.md (if it exists), the directory structure, and package manifests — not hallucinated. If neither context doc exists, the skill scans the codebase directly (directory structure, package manifests, entry points, README). After the overview, present exploration options tailored to what the project actually has (see R-002).

- **R-002** [integration]: The exploration menu after the overview is driven by structural signals detected during the scan, not a fixed template. Offer only options that are relevant to the project. Detection rules: (a) Has HTTP handlers, router setup, or middleware → offer "request lifecycle"; (b) Has `cmd/` directory, CLI framework imports, or `main()` with arg parsing → offer "command flow"; (c) Has 3+ packages/modules with cross-imports → offer "component dependency diagram"; (d) Has input validation, auth checks, or sanitization → offer "trust boundaries"; (e) Has a public API surface (`pkg/`, exported types, `__init__.py` exports) → offer "API surface map". Always offer: "Deep dive into a specific component" (user names file/directory), "Generate full HTML export", and a free-text "Or name a file, directory, or concept: ___" escape hatch. Each option has a number for quick selection.

- **R-003** [integration]: When the user selects an exploration option or names a file/directory/concept, the skill produces: (a) a mermaid diagram appropriate to the question (sequence diagram for flows, graph for dependencies, classDiagram for type relationships), and (b) a prose walkthrough of the same information underneath the diagram (2-5 paragraphs). The prose explains what the diagram shows and why the connections exist, not just restating the boxes and arrows. After presenting, offer 2-4 natural follow-up options based on what was shown: "Want to drill into {component mentioned}?", "Want to see what calls {function}?", "Want to see the data model for {entity}?". Always include "Back to overview" and "Export to HTML" as options.

- **R-004** [integration]: Every diagram and prose section must include an uncertainty marker for connections the skill could not verify through direct code tracing. Format: a "Confidence" line after the prose stating what was traced vs inferred. Example: "Confidence: Traced 4/6 connections via imports. The connection between ConfigManager and ProxyCore could not be traced — may use file watching, environment variables, or an internal channel." Connections traced via Serena symbol references are marked "traced". Connections inferred from directory adjacency, naming conventions, or documentation are marked "inferred". If all connections are traced, state "All connections verified via code tracing."

- **R-005** [integration]: The HTML output file is at `.correctless/artifacts/cexplain-{timestamp}.html`. In terminal mode (R-018), the file is generated when the user selects "Export to HTML" — it snapshots everything explored so far. In HTML mode (R-018), the file is created on the first exploration step and updated incrementally after each step — "Export to HTML" in this mode is a no-op (the file already exists and is current; the skill confirms the path). Both modes produce the same HTML format: (a) a prominent disclaimer banner at the top: "This document was auto-generated by /cexplain on {date}. Diagrams and descriptions are best-effort based on static analysis — not guaranteed to be complete or accurate. Verify critical details against the source code."; (b) a table of contents linking to each section explored; (c) each diagram rendered as an inline mermaid block (using the mermaid.js CDN script) with its prose walkthrough below it; (d) the confidence markers from each section. The file must be fully self-contained — no external dependencies except the mermaid.js CDN script tag for diagram rendering.

- **R-006** [integration]: If `mcp.serena` is `true` in `workflow-config.json`, the skill uses Serena MCP for code analysis: `get_code_map` for project structure, `find_symbol` for locating types/functions, `find_referencing_symbols` for tracing call chains and dependencies, `search_for_pattern` for regex with symbol context. If Serena is unavailable or any Serena call fails, fall back silently to grep/read equivalents per the standard fallback table plus: `get_code_map` → directory listing (glob for source files) + package manifest reading + directory structure analysis (same approach as R-001's no-docs scan). Do not abort, do not retry, do not warn mid-operation. If Serena was unavailable during the session, note it once at the end: "Note: Serena was unavailable — fell back to text-based analysis. Diagrams may be less precise." The skill must produce useful output without Serena — grep-based import tracing, directory structure analysis, and file reading are the baseline.

- **R-007** [unit]: Mermaid diagram output must be valid mermaid syntax. Each diagram is wrapped in a fenced code block with the `mermaid` language tag. The skill uses the appropriate diagram type for the question: `sequenceDiagram` for request/command flows, `graph TD` or `graph LR` for dependency/component relationships, `classDiagram` for type hierarchies, `stateDiagram-v2` for state machines. No diagram exceeds 30 nodes — if the real structure is larger, aggregate into logical groups and note "N items collapsed into {group name}".

- **R-008** [integration]: The skill detects the project type from structural signals and adjusts its vocabulary and diagram focus. Detection is not a classifier — it checks for presence of signals: (a) HTTP handlers/routes → "web API/service" vocabulary, request lifecycle focus; (b) CLI entry points with arg parsing → "CLI tool" vocabulary, command flow focus; (c) Exported public types with no main → "library" vocabulary, API surface focus; (d) Auth/validation/sanitization patterns → security-aware, trust boundary focus; (e) Multiple package.json/go.mod/Cargo.toml → "monorepo" vocabulary, package dependency focus. Multiple signals can be active simultaneously (a web API that is also a CLI tool). The skill does not label the project type — it offers relevant visualizations based on detected signals.

- **R-009** [unit]: The exploration is stateless across sessions. Each `/cexplain` invocation starts fresh with a full project scan. No exploration state is persisted to disk between sessions. Within a session, the skill maintains context of what has been explored so the user can say "back to overview" or "export what we covered" without re-scanning.

- **R-010** [integration]: When the user asks to "deep dive into" a specific file, directory, or concept, the skill reads the target and produces: for a file — its purpose, key functions/types with one-line descriptions, what imports it and what it imports, and a mermaid diagram of its internal structure or its place in the call graph; for a directory — its purpose, key files with descriptions, internal dependency structure, and a mermaid component diagram; for a concept (e.g., "authentication", "caching") — the skill searches the codebase for relevant code, identifies the files and functions involved, and produces a cross-cutting view with a mermaid diagram showing how the concept is implemented across modules.

- **R-011** [integration]: `/cexplain` SKILL.md is added to `skills/cexplain/SKILL.md`, registered in `sync.sh` for both Lite and Full distributions, documented in `docs/skills/cexplain.md`, and added to the README skills table under "Observability" (alongside /cstatus, /chelp, /csummary, /cmetrics, /cwtf).

- **R-012** [unit]: Token usage is logged to `.correctless/artifacts/token-log-{slug}.json` with `skill: "cexplain"`, `phase: "exploration"`, `agent_role: "explain-agent"`, `total_tokens`, `duration_ms`, and `timestamp`. Appends to existing file or creates it.

- **R-013** [integration]: The prose walkthrough for each exploration answer must reference specific file paths and line numbers where key behaviors are implemented. Not "the auth module handles this" but "auth is handled in `pkg/auth/middleware.go:42` (the `Authenticate` function) which is called by the router setup in `cmd/server/main.go:87`". This grounds the explanation in verifiable code locations. If the skill cannot determine the line number, it references the file path without a line number rather than guessing.

- **R-014** [unit]: The HTML export includes a "Generated by" footer with the Correctless version, the date, and a list of what was explored (section titles). The mermaid.js script tag uses a pinned CDN version (not `latest`) to prevent rendering breakage from upstream changes. The HTML uses minimal inline CSS — no external stylesheets.

- **R-015** [integration]: When the user enters free text (naming a file, directory, or concept rather than selecting a numbered option), the skill resolves the input: (a) if it matches a file path → deep dive into that file; (b) if it matches a directory → deep dive into that directory; (c) if it matches a known symbol (function, type, constant) → show that symbol's context, callers, and callees; (d) if it matches a concept keyword → cross-cutting search; (e) if ambiguous → present the top 3 matches and ask the user to clarify. The skill does not fail on free text — it always produces a response or asks for clarification.

- **R-016** [integration]: The 30-node limit per diagram (R-007) is enforced by grouping. When the real structure exceeds 30 nodes, the skill creates named subgraphs that collapse related items. Each collapsed group shows its name and item count (e.g., "Utilities (12 files)"). The user can request to "expand {group name}" as a follow-up, which produces a new diagram focused on that group's internals.

- **R-017** [integration]: `/cexplain` works without `/csetup` having been run. If `workflow-config.json` does not exist, the skill skips MCP config checks (treats Serena and Context7 as unavailable) and proceeds with direct codebase scanning. It does not require or prompt for `/csetup` — this skill is intentionally low-ceremony for quick exploration of unfamiliar codebases.

- **R-019** [integration]: If the project scan produces fewer than 2 identifiable components, the skill presents what it found (even if minimal) and offers: "This project has limited structure I can detect. You can: (1) Name a specific file or directory to explore, (2) Describe what the project does and I'll search for it, (3) Generate a directory tree overview." The skill never says "I can't help" — it always offers a path forward.

- **R-018** [integration]: On first invocation, the skill asks the user their preferred output mode: (1) Terminal — mermaid code blocks rendered inline in the conversation, with prose underneath (default for interactive exploration); (2) HTML file — each exploration step generates/appends to a self-contained HTML file with rendered diagrams (better for sharing or when the terminal doesn't render mermaid). The choice applies for the duration of the session. In HTML mode, the file is written incrementally as the user explores, and the path is shown after each update. The user can switch modes mid-session by saying "switch to HTML" or "switch to terminal".

## Won't Do

- Persist exploration state across sessions — each invocation is fresh
- Update AGENT_CONTEXT.md or ARCHITECTURE.md — this is a read-only exploration tool
- Execute code, run tests, or modify files — read-only analysis only
- Support non-mermaid diagram formats (PlantUML, Graphviz) — mermaid is the standard in Claude Code
- Interactive HTML (clickable drill-down in the exported page) — the HTML is a static snapshot; interactivity lives in the conversation
- Auto-invoke on project setup — the user explicitly runs `/cexplain` when they want it

## Risks

- **Mermaid rendering complexity** — large codebases may produce diagrams that are technically valid mermaid but render poorly (overlapping nodes, unreadable text). Mitigation: R-007 caps at 30 nodes with grouping; R-016 allows expansion.
- **Grep-based tracing misses indirect connections** — without Serena, the skill can only trace explicit imports and function calls. Event-driven architectures, dependency injection, and runtime wiring will be missed. Mitigation: R-004 uncertainty markers make this visible; R-006 notes when Serena wasn't available.
- **CDN dependency in HTML export** — the mermaid.js CDN must be reachable to render diagrams in the exported HTML. Mitigation: R-014 pins the CDN version; offline rendering is a Won't Do for now.
- **False confidence in traced connections** — grep-based "tracing" may match a function name that exists in multiple modules, producing a wrong connection. Mitigation: R-004 distinguishes "traced" vs "inferred"; R-013 includes file paths so the user can verify.
