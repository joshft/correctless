# Agent Context — Correctless

> Last updated: 2026-04-18

## What This Project Does

Claude Code plugin framework that enforces a correctness-oriented development workflow. Ships as a single `correctless/` distribution with 28 skills and configurable intensity levels (standard, high, critical). Standard intensity (~10-15 min/feature) covers core TDD workflow; high/critical intensity (~1-2 hr/feature) adds formal modeling, adversarial review, and convergence auditing. The core principle: never let an agent grade its own work — each workflow phase uses a separate agent with a different lens.

## Design Patterns

- **Source-to-dist sync** (PAT-001): edit in `skills/`, `hooks/`, `templates/`, `helpers/` only — run `sync.sh` to propagate to `correctless/`
- **Agent separation** (PAT-002): each TDD phase (RED/GREEN/QA/mini-audit) is a different agent — enforced via `context: fork` and sub-agent spawning
- **Phase-gated writes** (PAT-003): `workflow-gate.sh` blocks file operations that violate the current phase (RED blocks source, QA blocks everything)
- **Branch-scoped state** (PAT-004): state lives in `.correctless/artifacts/workflow-state-{branch-slug}.json` — `workflow-advance.sh` is the only writer. Includes spec integrity fields (`spec_hash`, `spec_line_count`) for detecting post-review spec mutations
- **Effective intensity** (PAT-005): each pipeline skill and gated skill computes `max(project_intensity, feature_intensity)` using ordering `standard < high < critical`. Project intensity from `workflow.intensity` in config, feature intensity from `workflow-advance.sh status`. Fallback: feature_intensity → workflow.intensity → standard
- **Shift-left review** (PAT-006): `/creview` and `/creview-spec` read historical findings (QA, Olympics audits, Devil's Advocate reports) to detect recurring patterns. Classification is ephemeral — see ABS-002 in ARCHITECTURE.md. 10-file budget (PAT-004)
- **Cross-skill calibration** (ABS-005): `/cverify` writes outcome data (QA rounds, BLOCKING findings, actual_tokens) to `.correctless/meta/intensity-calibration.json`, `/cspec` reads it during intensity detection as a post-signal modifier. Token data (ABS-006) enriches calibration — features exceeding 200K tokens trigger auto-raise alongside QA rounds >= 3 and findings >= 8. cspec is read-only — only cverify writes. 50-entry recency window. Configurable modes: passive (advisory), active (auto-raise), hybrid (passive→active after 5 entries)
- **Semi-auto mode** (PAT-009): `/cauto` orchestrates the implementation pipeline after human-approved spec review. Pipeline: `ctdd → simplify → cverify → cupdate-arch → cdocs → consolidation → PR`. Accepts any active workflow phase via a fixed phase-to-step mapping (flexible entry), validates skipped-phase artifacts before resuming, runs scoped commit-and-push consolidation before PR creation (TB-004c), and prints a structured end-of-pipeline summary with findings, phase breakdown, and artifact paths. Escalates on architectural decisions, persistent failures, or spec contradictions. Preferences from `.correctless/preferences.md` guide autonomous decisions. See ABS-007 (escalation file), ABS-008 (preferences contract), TB-004 (orchestrator autonomy boundary), TB-004c (scoped git push authority).
- **Auto Mode Phase 2** (ABS-011..017, PAT-011): Policy-driven decision engine extending `/cauto` with tiered architecture (Tier 0 policy engine, Tier 1 worker self-resolution, Tier 2 ephemeral decision agents, Tier 3 lightweight supervisor, Tier 4 hard stop). Decisions use structured DR-xxx format routed through the tier hierarchy. Budget enforcement (token + time) with warn/hard-stop thresholds. Decision record (append-only DD-xxx entries with size-regression detection). Intent summary (immutable, SHA-256 hash verified). Auto Run Report generated on completion/pause. Policy integrity verified via hash chain (PAT-011). See `scripts/auto-policy.sh`, `scripts/decision-routing.sh`, `agents/supervisor.md`, `agents/decision-agent.md`.
- **Auto Mode Phase 3** (ABS-018..020, TB-004a): Extends `/cauto` to orchestrate from prompt to PR — spec writing, autonomous review with supervisor-triaged findings, mandatory human spec approval, then Phase 2 pipeline. Supervisor mandate expansion: configurable levels (conservative/moderate/aggressive), architectural decisions with spec citation enforcement, 7 hard-limit conditions bypass supervisor to Tier 4. Override window scrutiny: 3-phase supervisor review (issuance → per-action → closure) with base-commit cross-check, mechanical file-touch drift detection, spec completeness verification, Jaccard-similarity retry prevention, and cross-run override pattern detection (ABS-021 — preserved logs in `.correctless/meta/overrides/`, 50-file cap, Jaccard >= 0.4 threshold triggers escalation when 2+ recent runs share a reason). `/cauto` preserves override logs on every terminal state (Step 9.5). `/cmetrics` surfaces override health (mean per run, reason clusters, elevated rate warning). See `scripts/review-triage.sh`, `scripts/supervisor-mandate.sh`, `scripts/override-scrutiny.sh`, `scripts/override-crosscheck.sh`.
- **Path-scoped rules** (ABS-009): rule files under `.claude/rules/*.md` with YAML `paths:` frontmatter load into an agent's editing context whenever a scoped file is opened. PAT-001 (PreToolUse hook fail-closed posture) lives in `.claude/rules/hooks-pretooluse.md`; `.correctless/ARCHITECTURE.md` contains only the 2-line index entry. Structural drift is enforced by `tests/test-architecture-drift.sh`. **Provisional** — the convention is on probation until a post-merge measurement gate (MG-001/MG-002/MG-003) confirms the rule-in-context prevention signal. Until then, new PAT entries land full-body in ARCHITECTURE.md. See `.correctless/specs/path-scoped-rules-pat001.md` for scope and rollback procedure.
- **Integration test contracts** (ABS-024): `/cspec` writes Entry/Through/Exit constraints on `[integration]` rules, derived from ARCHITECTURE.md entrypoints (ABS-023). `/ctdd` test audit verifies contracts with tiered severity (Entry=mechanical BLOCKING, Through=semi-mechanical BLOCKING/UNCERTAIN, Exit=semantic BLOCKING/ADVISORY). Contracts tell the TDD agent what to exercise and what not to mock — see `skills/cspec/SKILL.md` Step 4a and `skills/ctdd/SKILL.md` test audit section
- **Entrypoint-aware TDD** (Phase 1): The RED phase test agent reads ARCHITECTURE.md entrypoints and key patterns (Key Patterns, Layer Conventions, Trust Boundaries) before writing integration tests. Tests are written through documented entrypoints via `test_via` patterns, not by importing internal packages directly. The test audit's check 10 (internal import bypass detection) catches violations — language-aware for Go, TS/JS, Python, Rust. Graceful fallback when no entrypoints exist. See `skills/ctdd/SKILL.md` Phase RED and test audit sections
- **MCP integration** (optional): Serena for symbol-level code analysis, Context7 for library docs — check `mcp.serena` and `mcp.context7` in workflow-config.json. Falls back to grep/read silently when unavailable

## Common Pitfalls

- **Editing distribution target directly**: changes in `correctless/` will be overwritten by `sync.sh` — **Instead**: edit in root `skills/`, `hooks/`, `templates/`, `helpers/` then run `bash sync.sh`
- **Forgetting to sync after edits**: distribution targets go stale — **Instead**: always run `bash sync.sh` after editing source files, then `bash tests/test.sh` to verify
- **Adding a skill without updating sync.sh**: new skills won't appear in the distribution — **Instead**: add the skill directory to the skill list in `sync.sh`
- **Inline subagent prompts in skill files**: defining a subagent as prose inside a `skills/*/SKILL.md` file is brittle — the prompt can't actually be invoked without `Task()` wiring, and drift between the prose and any real implementation is invisible — **Instead**: define the subagent in `agents/{name}.md` (frontmatter + system prompt body) and invoke via `Task(subagent_type="correctless:{name}")`. See ABS-010 in `.correctless/ARCHITECTURE.md` for the contract and `agents/fix-diff-reviewer.md` for the canonical example

| Run tests | `bash tests/test.sh && bash tests/test-mcp.sh` (or run all: see workflow-config.json `commands.test`) |

| Configure auto-mode policy | `.correctless/config/auto-policy.json` |
