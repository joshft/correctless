# Spec: Override Frequency Metrics — Preserve Override Logs and Surface Gate Health

## Metadata
- **Task**: override-freq-metrics
- **Recommended-intensity**: high
- **Intensity**: high
- **Intensity reason**: file path signal (hooks/, scripts/override-scrutiny.sh); keyword signal (override, security); antipattern signal (AP-023 directly motivated this feature)
- **Override**: none

## What

Preserve override logs from ephemeral branch artifacts to a persistent location so that override frequency can be tracked across pipeline runs. Surface override frequency in `/cmetrics` as a gate health indicator. Enable the Phase 3 supervisor to detect cross-run override patterns during override issuance review — when the same override reason recurs across multiple runs, the supervisor escalates as a probable gate bug rather than approving individually.

## Rules

- **R-001** [unit]: When `/cauto` completes — whether successfully (PR created), via escalation (pipeline halted), or on any other terminal state — it reads the override log from `.correctless/artifacts/override-log.json`, filters entries to only those whose `branch` field matches the current branch name (the override log is a shared file, not per-branch — entries from other features must be excluded), and writes the filtered entries to `.correctless/meta/overrides/{task-slug}-{YYYYMMDD}.json`. The date suffix prevents collision when the same task-slug is reused across features. The preserved file contains the filtered array plus a metadata wrapper: `{"task_slug": "{task-slug}", "branch": "{branch}", "completed_at": "{ISO timestamp}", "override_count": N, "overrides": [...filtered array...]}`. If the override log does not exist, or no entries match the current branch, write the metadata wrapper with `"override_count": 0` and `"overrides": []` — zero-override runs are data, not absence of data. Prerequisite: add `Write(.correctless/meta/overrides/*)` to cauto's allowed-tools frontmatter.
- **R-002** [unit]: `/cdocs` includes override count in the workflow-history.md entry. The existing entry format `"Branch: {branch}. Rules: {count}. QA rounds: {N}. Findings fixed: {N}. {description}."` is extended to include `"Overrides: {N}."` after `"Findings fixed: {N}."` if the override count is greater than 0. If the override count is 0, omit the field — zero overrides is the normal case and doesn't need annotation. `/cdocs` reads the override count from the preserved file at `.correctless/meta/overrides/{task-slug}-*.json` (most recent by filename sort if multiple exist per R-001), falling back to the ephemeral override log at `.correctless/artifacts/override-log.json` (filtering by current branch) if no preserved file exists (pipeline ran without `/cauto`). If neither file exists, the override count is 0.
- **R-003** [unit]: `/cmetrics` adds an "Override Health" section to the dashboard after the "Velocity" section. The section reads all files in `.correctless/meta/overrides/*.json`, computes: (a) total overrides across all preserved runs, (b) mean overrides per run (total / count of preserved files), (c) override reason frequency — group overrides by reason using Jaccard similarity (threshold 0.3, invoked via `Bash(source scripts/override-scrutiny.sh && jaccard_similarity "reason_a" "reason_b")`) to cluster similar reasons, then show up to 3 clusters sorted by count descending, ties broken alphabetically by shortest reason in the cluster. The clustering threshold (0.3) is intentionally lower than the retry-prevention threshold (0.4 in PRH-006) — clustering for reporting should be loose to surface patterns, while retry prevention should be conservative to avoid blocking legitimate overrides. If mean overrides per run > 0.5, emit a warning: "Override rate is elevated ({mean}/run). Check the top reasons — this may indicate a gate misclassification (AP-023)." If `.correctless/meta/overrides/` doesn't exist or is empty, the section says: "No override data yet. Override tracking starts automatically on the next `/cauto` run."
- **R-004** [integration]: When the Phase 3 supervisor activates on `override_issued` (INV-035 in auto-mode-phase-3 spec), the `review_override_issuance` function in `scripts/override-scrutiny.sh` reads `.correctless/meta/overrides/*.json` and checks whether the incoming override reason matches reasons from recent runs. "Recent" is defined as: last 10 preserved files sorted by `completed_at` timestamp. Matching uses `jaccard_similarity` (already available in the same file) with threshold 0.4 (same as retry prevention — conservative for an escalation trigger). If 2+ recent runs contain an override with a matching reason, `review_override_issuance` returns `escalate_to_human` with a structured message that makes the cross-run pattern visible: "CROSS-RUN OVERRIDE PATTERN DETECTED: This override reason has recurred in {N} of the last {M} runs ({list of task-slugs with dates}). This pattern suggests a gate misclassification (AP-023), not an exceptional condition. Fix the underlying gate condition rather than continuing to override." The message includes the matching task slugs and dates so the human can verify the pattern without digging through logs. This cross-run check runs BEFORE the existing per-run scrutiny (base-commit cross-check, retry prevention). If the cross-run check escalates, the per-run checks are skipped — the escalation is sufficient. If the human approves the override despite the cross-run pattern, the override proceeds through the normal three-phase lifecycle (issuance → per-action → closure) as if it were a fresh override. Prerequisite: update ABS-020 in ARCHITECTURE.md to document the cross-run pre-check and its short-circuit behavior.
- **R-005** [unit]: Override data lives in `.correctless/meta/overrides/` and is intentionally local-only (not version-controlled). The data persists across branches because `.correctless/meta/` is a project-level directory (not branch-scoped like `.correctless/artifacts/` filenames). In the Correctless development repo, `.correctless/meta/` is gitignored — override data follows this existing policy, matching the treatment of `intensity-calibration.json` and `drift-debt.json`. Override reasons may contain internal project context (gate names, phase details, debugging notes) that shouldn't be committed. For team-wide cross-run pattern detection, each developer's supervisor detects patterns in their own local data — shared detection would require a separate mechanism (out of scope). Prerequisite: add ABS-021 to ARCHITECTURE.md documenting the `meta/overrides/` directory contract (sole writer: `/cauto`, readers: `/cmetrics`, `/cdocs`, `override-scrutiny.sh`, retention: 50-file cap, schema: metadata wrapper with task-slug keying).
- **R-006** [unit]: Preserved override files in `.correctless/meta/overrides/` are capped at 50 files. When `/cauto` writes a new preserved file (R-001) and the directory already contains 50+ files, it deletes the oldest file(s) by `completed_at` timestamp (read via `jq -r '.completed_at'` from each file). Files where `completed_at` is missing or not a valid ISO timestamp are treated as oldest (evicted first). Log a warning when evicting a malformed file. This prevents unbounded growth while retaining enough history for meaningful pattern detection.

## Won't Do

- **Broader artifact preservation** — only override logs. Other ephemeral artifacts (decision records, audit trail, token log) stay branch-scoped. If `/caudit` needs decision-record history later, that's a separate spec with a concrete use case.
- **`/caudit` heuristic integration** — override hotspots as audit targets is speculative without a concrete case where it would have helped. Defer.
- **Override reason normalization** — Jaccard similarity handles natural language variation well enough. Building a controlled vocabulary of override reasons would be premature.
- **Real-time alerting** — no webhook, no Slack notification when override rate exceeds threshold. `/cmetrics` is pull-based (human checks it) and supervisor is event-driven (fires on override issuance). Push-based alerting is overkill for a single-developer workflow.

## Risks

- **Jaccard similarity false positives on short reasons**: Short override reasons ("gate bug", "spec edit") have few tokens, making Jaccard noisy. Two genuinely different 2-word reasons might cluster together.
  1. Accept (recommended) — the threshold (0.4 for escalation, 0.3 for reporting) is already calibrated from PRH-006 retry prevention. If false positives surface, raise the threshold. The cost of a false cluster in `/cmetrics` is a misleading "top reason" — annoying but not dangerous. The cost of a false escalation from the supervisor is a human seeing "this reason recurred" when it didn't — they dismiss it and proceed. Neither is high-stakes.

- **Legitimate recurring overrides trigger false escalations**: If 30% of features legitimately need overrides for the same structural reason (e.g., a gate condition that's correct but awkward for certain feature shapes), the cross-run check (R-004) escalates on every run and the supervisor becomes a bottleneck.
  1. Accept (recommended) — the escalation is to a human, not a block. The human sees "this pattern recurred in N runs" and either (a) fixes the underlying gate if it's actually miscalibrated, or (b) dismisses the escalation if the overrides are genuinely legitimate. If the same reason keeps getting dismissed, that's a signal to add a gate exception for that specific case. The 0.4 threshold is conservative enough to avoid clustering genuinely different reasons.

- **Phase 3 supervisor not active for most runs**: The supervisor integration (R-004) only fires when Phase 3 is active. For Phase 1/2 runs (which are all current runs), the cross-run pattern detection doesn't fire. The gate bug we just fixed would NOT have been caught by R-004 during the session because Phase 3 wasn't active.
  1. Accept (recommended) — `/cmetrics` (R-003) catches the pattern for Phase 1/2 runs. R-004 is defense-in-depth for when Phase 3 is active. The two mechanisms cover different operational modes.
