# Spec: /cauto UX Improvements — Flexible Entry, Consolidated PR, Pipeline Summary

## Metadata
- **Task**: auto-ux-improvements
- **Recommended-intensity**: standard
- **Intensity**: high
- **Intensity reason**: project floor (workflow.intensity=high); no security signals triggered
- **Override**: none

## What

Improve `/cauto` usability based on the first real pipeline run (scanner-expansion, 2026-04-13). Three changes: (1) replace the rigid `review`/`review-spec` phase gate with a phase-aware resume that accepts any workflow phase and runs the remaining pipeline steps, validating artifacts from prior phases before skipping them; (2) add a commit-and-push consolidation step before PR creation so all pipeline artifacts (docs, architecture, verification report) are included in the PR; (3) add a structured end-of-pipeline summary showing findings, decisions, deferrals, and per-phase token/cost/time breakdown.

## Rules

- **R-001** [unit]: `/cauto` accepts any active workflow phase, not just `review`/`review-spec`. Given the current phase, it computes the remaining pipeline steps using a fixed phase-to-step mapping: `review`/`review-spec` → full pipeline (`ctdd` through PR); `tdd-tests`/`tdd-impl`/`tdd-qa` → resume from `ctdd` (which handles internal TDD phases); `done` → `simplify` through PR; `verified` → `cupdate-arch` (if high+) through PR; `documented` → PR only. Phases `spec` and `model` are rejected — the spec must be reviewed before the implementation pipeline can run. The `spec` rejection message says: "Run `/creview` or `/creview-spec` first — /cauto starts after spec review." This phase-to-step mapping applies to the implementation pipeline only. When no workflow exists, `/cauto` delegates to Phase 3 entry logic (INV-019), which transitions through spec creation and review before entering R-001's mapping at `review`/`review-spec`. Mid-TDD resume semantics are delegated to `/ctdd` — it has its own internal phase-aware logic and will not restart from `tdd-tests` if the workflow is already at `tdd-impl` or `tdd-qa`. `/cauto` invokes `/ctdd` and trusts it to handle the internal TDD state machine from wherever it currently is.
- **R-002** [unit]: Before skipping a phase that is *behind* the current phase (already completed), `/cauto` validates that the phase's artifacts exist. For the current phase, `/cauto` invokes the corresponding skill directly without validation — validation only applies to phases being skipped, not phases being resumed. Validation checks per phase: (a) `ctdd` complete → test suite passes; (b) `simplify` → no validation needed (optional step); (c) `cverify` complete → verification report exists at `.correctless/verification/{task-slug}-verification.md`; (d) `cupdate-arch` → no validation needed (advisory); (e) `cdocs` → no validation needed (advisory). For (a), `/cauto` executes `commands.test` from workflow-config.json using the Bash tool's `timeout` parameter, defaulting to 300 seconds. The timeout is configurable via `commands.test_timeout` in workflow-config.json (integer, seconds). Exit code 0 = validation pass. Non-zero or timeout = validation fail; `/cauto` re-runs the phase. If validation fails 2 consecutive times, skip validation and proceed — the phase's own skill will re-verify. Note: the test-suite-pass check catches "implementation broke" cases cleanly but only partially catches "user modified state outside the pipeline" cases — spec-to-implementation alignment is `/cverify`'s responsibility, which runs next. The validation failure is logged in the audit trail with type `artifact_validation_failed`.
- **R-003** [unit]: Between `/cdocs` completion and PR creation, `/cauto` runs a consolidation step (F-001: scoped staging to prevent accidental commit of secrets or unintended files). The step: (1) stage only known pipeline output paths: files changed on the branch (`git diff main...HEAD --name-only`), plus explicit pipeline output paths (`.correctless/verification/{task-slug}-verification.md`, `.correctless/ARCHITECTURE.md`, `.correctless/AGENT_CONTEXT.md`, `README.md`, `CONTRIBUTING.md`, `docs/workflow-history.md`, `docs/features/*.md`). Unknown untracked files are never staged. The staging set is the union: committed changes on the branch (from the diff) plus uncommitted pipeline outputs matching the explicit path list. The explicit path list is a constant in the implementing script — future additions to pipeline outputs must be added here. (2) Verify nothing under `.correctless/artifacts/` was staged — if anything was, unstage it with `git reset HEAD .correctless/artifacts/` (belt-and-suspenders for gitignore misconfiguration). (3) Commit with message `"Add pipeline artifacts for {task-slug}"`. (4) Push — derive remote name from `git config --get branch.$(git branch --show-current).remote` for tracked branches; if unset (fresh branch), use the first remote from `git remote | head -1` with `--set-upstream`; if `git remote` returns nothing, abort with: "No git remote configured. Push manually or add a remote." R-003 must not push to branches matching `main`, `master`, `develop`, or `release/*` — abort with an error if the current branch matches any of these. If there are no uncommitted changes after step 1-2, skip steps 3-4 (no-op). If `git push` fails (auth failure, rejected push), `/cauto` aborts with a clear error message including the git error output. The local commit is preserved but the PR step is skipped. Re-running `/cauto` after fixing the push issue resumes at the PR step (the phase is `documented`, so R-001 maps to PR-only).
- **R-004** [unit]: After PR creation (or after the last pipeline step if `pr_creation: skip`), `/cauto` prints a structured summary. The summary has three sections: (a) **Findings & Decisions** — every QA finding, verification finding, review decision, override, and architectural decision from the pipeline, each with its disposition (fixed, deferred, accepted). Includes deferred items with their reason. Items from sources without a severity field (verification findings, override activity) are always shown inline — they are not subject to severity-based truncation. (b) **Phase Breakdown** — a table with one row per pipeline step showing: step name, duration, token count, and result (pass/fail/skipped/reverted/incomplete). Duration = last `skill_completed` elapsed_ms − first `skill_started` elapsed_ms for that skill name (clock-independent, uses audit trail's authoritative time representation). Phases that never completed (pipeline aborted mid-run, detected by `skill_started` without matching `skill_completed`) appear with duration up to the abort point and result `(incomplete)`. The orchestrator logs `skill_started`/`skill_completed` for all pipeline steps including `/simplify`, which does not log its own entries. (c) **Artifacts** — file paths for: spec, verification report, QA findings file, audit trail, and PR URL. Documentation and architecture changes get a one-line summary each (e.g., "README, AGENT_CONTEXT updated; PAT-014, PAT-015 added"). If the Findings & Decisions section has more than 20 items from severity-bearing sources, truncation applies: show inline (a) all items with severity HIGH or CRITICAL, (b) all items with explicit disposition `deferred` (user needs to see what got punted regardless of severity), (c) all override activity, (d) all items from non-severity sources (verification findings). Items not matching any of these four criteria go into the count-and-reference summary (e.g., "QA Findings: 47 total — 3 HIGH + 2 deferred shown below, see qa-findings-{task-slug}.json for full list").
- **R-005** [unit]: The audit trail event type list is extended with `artifact_validation_failed` (from R-002). The event includes `phase`, `expected_artifact`, and `validation_error` fields alongside the standard `type`, `timestamp`, `skill`, `elapsed_ms`. This type must also be added to the canonical event type list in `skills/cauto/SKILL.md`'s Audit Trail section (currently 7 event types).
- **R-006** [unit]: The summary's Findings & Decisions section reads from these sources: (a) `.correctless/artifacts/qa-findings-{task-slug}.json` for QA findings and their dispositions; (b) `.correctless/verification/{task-slug}-verification.md` for verification findings (parse the Rule Coverage and Smells sections); (c) `.correctless/artifacts/review-decisions-{task-slug}.json` for review triage decisions (if Phase 3 review ran); (d) `.correctless/artifacts/override-log-{branch-slug}.json` for override activity (if any overrides were issued); (e) the audit trail JSONL (`.correctless/artifacts/audit-trail-{branch-slug}.jsonl`) for escalation events, simplify reverts, and architectural decision events. If a source file doesn't exist, that source is omitted from the summary (not an error). Note: QA findings and verification report use `{task-slug}` (the feature name from workflow init). Override log, audit trail, and token log use `{branch-slug}` (derived from the branch name via `branch_slug()` in scripts/lib.sh). These are different values — task-slug is e.g. `scanner-expansion`, branch-slug is e.g. `feature-scanner-expansion-0c9277`.
- **R-007** [unit]: The summary's Phase Breakdown table uses **skill names** (ctdd, simplify, cverify, cupdate-arch, cdocs) as row identifiers, not workflow phase names (tdd-tests, tdd-impl, etc.). This aligns with both the audit trail entries (which use skill names in the `skill` field) and the token log entries (which use skill names in the `skill` field). No phase-to-skill mapping is needed — the table shows what was invoked, which is what humans care about. Duration per row = last `skill_completed` elapsed_ms − first `skill_started` elapsed_ms for that skill name. For skills with multiple attempts (retries, QA rounds), the span covers all attempts. Token count per row is computed by summing `total_tokens` from `.correctless/artifacts/token-log-{branch-slug}.jsonl` entries where the `skill` field matches the row's skill name. If the token log doesn't exist or has no entries for a skill, the token column shows `—`. Note: R-007 uses skill names for the summary display; R-001 and R-002 use phase names for workflow transitions and validation. These are different namespaces serving different audiences (user-facing summary vs. workflow state machine).

## Won't Do

- **Changing individual skill behavior** — each skill (`/ctdd`, `/cverify`, etc.) is untouched. Only the orchestrator's entry logic, commit handling, and summary generation change.
- **Automatic re-running of all prior phases on resume** — if the user is at `done`, `/cauto` trusts that TDD completed (validated by test suite pass per R-002). It does not re-run `/ctdd` from scratch.
- **Interactive summary** — the summary is printed once at the end. It's not a wizard or interactive triage. The user reads it and decides what to do.
- **Cost estimation before pipeline starts** — the per-phase cost breakdown is retrospective only.
- **Changes to preferences.md schema** — existing preferences contract (ABS-008) is unchanged.
- **Modifying `/ctdd` resume logic** — `/ctdd` already handles being invoked at any internal TDD phase (tdd-tests, tdd-impl, tdd-qa). R-001 delegates mid-TDD resume to `/ctdd` rather than reimplementing it.
- **Phase 3 spec-creation flow** — R-001's phase mapping begins after Phase 3 transitions through spec creation and review. Phase 3's INV-019 no-workflow entry mode is not modified by this spec.

## Risks

- **Stale phase with passing tests**: A user manually resets the phase to `done` after partially completing TDD. Tests pass (because some were written) but not all spec rules are covered. R-002 validates test pass but can't verify rule coverage without re-running `/cverify`. Similarly, if the user added features the spec doesn't cover, the TDD output has diverged from the spec — R-002 won't catch this, but `/cverify` will when it runs next.
  1. Accept (recommended) — R-002 validates the mechanical artifact (test suite pass). Spec-to-implementation alignment is `/cverify`'s job, which will run next in the pipeline and catch gaps. Defense in depth: R-002 catches broken code, `/cverify` catches spec drift.
  2. Mitigate — add rule-coverage check to R-002's `ctdd` validation. Heavyweight for a phase-skip check.

- **Token log missing or incomplete**: If the token-tracking hook wasn't active for some phases, the Phase Breakdown table has gaps.
  1. Accept (recommended) — R-007 already handles this with `—` for missing data. The audit trail `elapsed_ms` always exists (orchestrator writes it), so duration is always available even if tokens aren't.

- **Large summary for complex features**: A feature with 3 QA rounds and 15+ findings produces a long Findings & Decisions section.
  1. Mitigated — R-004 caps inline display at 20 items for severity-bearing sources, showing HIGH/CRITICAL + deferred + override + non-severity items inline with a count and file reference for the rest.
