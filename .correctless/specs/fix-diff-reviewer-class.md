## Context

Add a **class-shaped bug detection lens** to the fix-diff-reviewer agent at `agents/fix-diff-reviewer.md`. When the bug under review is class-shaped — describes a pattern that could have multiple instances in the same file or module — the reviewer must grep for sibling instances before approving. The reviewer's existing pinned tool surface (Read, Grep, Glob) already supports this work; only the prompt direction was missing. PR #124 (2026-05-14) approved a fix at the outer `collect_artifacts` boundary in `scripts/build-dashboard.sh` and never checked the inner `read_file_json` helper using the same `--arg content "$content"` pattern. One month later, PMB-019 was the recurrence. The lens is preventive structure for the class of "same script, same shape, fixed at the wrong scope" failures.

## Scope

**In scope.** A new lens added to `agents/fix-diff-reviewer.md` between the existing "What to check for each hunk" list and the "Output contract" section. The lens uses a two-signal detection: **primary** is the diff content itself (the reviewer inspects diff text and surrounding hunk context for patterns that suggest a scope-narrowed instance fix), **refinement** is the round-level finding list passed via a new `<UNTRUSTED_FINDING_DESCRIPTION>` fence emitted in `/caudit` **Step 6a**'s per-round prompt-assembly block (the sub-step that wraps `<UNTRUSTED_RULES>` and `<UNTRUSTED_DIFF>`; the reviewer is invoked once per round, not per finding — verified against the Step 6a per-round Task invocation in `skills/caudit/SKILL.md`). Either signal can trigger the lens; both together raise confidence. When triggered, the reviewer is instructed to grep the file under fix (and any sibling module within a **bounded scope: same directory + same language extension**, expressed as a CLOSED allow-list per CS-015, never `.env*` / `.correctless/preferences*` / `.correctless/artifacts/autonomous-decisions-*` / `.git/objects/**`) for instances of the same pattern. Unaddressed siblings — when the diff does NOT include a machine-checkable `SIBLING-DEFERRED:` marker enumerating them with substantive rationale — become a HIGH finding routed through the existing JSON output contract.

Empty/whitespace-only descriptions are omitted from the array (treated identically to no description, RS-022). When the array would be empty, the entire fence is omitted (no empty fence emission). The reviewer's lens still degrades gracefully when the fence is absent — diff signal alone — and emits a one-line advisory on the degradation path (RS-031).

**Per-fence size bound (RS-007).** The per-description size is capped at 4 KB after which truncation appends a marker (`[truncated: N more bytes]`). The aggregate fence is capped at 16 KB carved from /caudit's 100 KB ceiling AFTER DIFF+RULES are measured (CS-014, RS-013); when the array would exceed the carved budget, individual descriptions are truncated proportionally — preserving a marker entry per finding — before any entry is dropped. ### CS-002: Two-signal detection — diff content primary, finding description refinement (prose layer)
- **Type**: must
- **Category**: functional
- **Statement**: The lens section instructs the reviewer to detect class-shape using two signals, with **two distinct seed inputs mapped to two distinct signals (RS-016)**: **(a) primary — diff content**: examine the diff text and surrounding hunk context for patterns that suggest a scope-narrowed instance fix. The prose includes a non-exhaustive seed list of **code patterns** the reviewer can recognize in the diff — e.g., `--arg "$var"` substituted with `--rawfile`/`--slurpfile`, `2>/dev/null` additions at one error site, single-site `lock`/`unlock` pairs, single-site loop-variable scope fixes. This is the CS-002(a) **code-pattern** seed list. **(b) refinement — finding description**: when the `<UNTRUSTED_FINDING_DESCRIPTION>` fence is present (passed by /caudit step 6a in JSON-array form per CS-011), examine each finding's wording for class-shape indicators. This is the CS-002(b) **bug-description keyword** seed list (e.g., "overflow", "exhaust", "race", "deadlock", "truncate" — the list PRH-003 governs). The two seed lists are separate inputs to separate signals and MUST NOT be conflated (resolves the CS-002-vs-PRH-003 contradiction, RS-016: code-pattern seeds anchor the diff signal; keyword seeds anchor the description signal). Either signal can trigger the lens; both together raise confidence. Both seed lists are explicitly marked non-exhaustive in the prose. **This invariant is the prose-layer test — CS-013 adds a prompt-composition test layer that constructs synthetic /caudit Step 6a prompts from real fixtures and asserts the assembled text carries the trigger conditions, because substring-grep over the agent file alone cannot verify the assembled prompt is shaped correctly (RS-001).**
- **Boundary**: TB-005 (both signals are untrusted-data inputs; the reviewer applies its own semantic judgment over them)
- **Violated when**: only one seed list is present; the code-pattern seeds and keyword seeds are conflated into a single undifferentiated list; the seed lists are not marked non-exhaustive; the lens depends on `<UNTRUSTED_FINDING_DESCRIPTION>` being present (must degrade gracefully when absent per PAT-019)
- **Enforcement**: CI test assertion (heading + semantic-test phrase + code-pattern seed phrase presence on the diff signal + keyword seed phrase presence on the description signal + non-exhaustive marker on BOTH + graceful-degradation language for absent finding-description fence). **Composed with CS-013 prompt-composition tests** — CS-002 enforces prose composition, CS-013 enforces prompt-composition under real fixtures.
- **Guards against**: AP-024 (hardcoded list goes stale) — the harm mode where the keyword enumeration is class-incomplete by construction
- **Test approach**: unit (prose composition) — paired with CS-013 prompt-composition tests
- **Risk**: high — the harm mode the user flagged (keyword enumeration becoming class-incomplete) materializes silently
- **Implemented in**: (GREEN phase)

### CS-003: Reviewer must grep for sibling instances when the lens is triggered (proximity-anchored, anti-negative)
- **Type**: must
- **Category**: functional
- **Statement**: When the lens is triggered, the prose explicitly directs the reviewer to grep the file under fix and bounded sibling modules (CS-015 scope) for instances of the same pattern. The directive is a **positive imperative sentence** binding "grep" and "sibling" within a single line (≤120 chars), names at least two of the pinned tools `Read`/`Grep`/`Glob` on the same line, and is NOT a hedged or negative sentence.
- **Boundary**: TB-005
- **Violated when**: the sibling-grep step is absent; the directive does not name the tools; "sibling" appears without "grep" within the section body; the matched line begins with negation tokens; the imperative is hedged by a HEDGE_TOKEN
- **Enforcement**: CI test assertion — section body contains a single ≤120-char line matching `\b(grep|search) .{0,80}\b(sibling|other instances|same pattern)\b` AND the matched line names ≥2 of `{Read, Grep, Glob}` AND the matched line's FIRST token is one of `{grep, search}` (mechanical proxy for "imperative form" — replaces the ungreppable grammar judgment, RS-027) AND the matched line does NOT begin with any negation token AND no token from the enumerated `HEDGE_TOKENS` reject-list (`may`, `might`, `consider`, `could`, `if confident`, `optionally`, `where appropriate`, `should consider`, `you may want to`) precedes the imperative on that line. Closes RS-012 (negation) and RS-027 (hedge tokens + imperative definition).
- **Guards against**: AP-035 at the call-chain level (gate runs but does not look beyond the failing scope); negation-bypass and hedge-bypass classes
- **Test approach**: unit
- **Risk**: high — this directive is the actual behavior change; absence reduces the lens to a slogan
- **Implemented in**: (GREEN phase)

### CS-004: Enumeration carve-out — `SIBLING-DEFERRED:` marker in the diff (with non-exhaustive comment styles)
- **Type**: must
- **Category**: functional
- **Statement**: The lens body specifies a machine-checkable enumeration carve-out: when the diff contains one or more comment lines matching the regex `SIBLING-DEFERRED:\s+\S+(:\d+)?\s+[—-]\s+.+` (literal token `SIBLING-DEFERRED:`, then file-path with optional `:line-number`, separator, then rationale prose — the line-number is genuinely optional, parenthesized in the regex) AND the marker covers each sibling instance the reviewer identifies, AND the marker validity contract (CS-016) is satisfied, the reviewer approves the marker-covered siblings.

  The marker may live in **true syntactic comment forms used in the project's source files (non-exhaustive — examples: `#` (bash/Python/YAML/TOML), `//` (JS/TS/Go/C-family), `--` (SQL/Lua), `/* */` (C-family/CSS), `<!-- -->` (HTML/Markdown/XML), `;` (INI/Lisp/Assembly))**. Python triple-quoted strings (`"""..."""`) are NOT listed as a comment style — they are string literals (sometimes USED as docstrings) and listing them would collide with the marker-in-string-literal bypass class (RS-011 / CS-016d adversarial fixtures). Markers MUST be at the start of a true comment, not inside a string literal value. The prose explicitly states the comment-style list is non-exhaustive.

  **Marker-format migration (RS-029).** SIBLING-DEFERRED markers are durable on-disk comments. The prose states that any FUTURE tightening of this marker regex MUST either (i) accept the prior format for a deprecation window OR (ii) ship a one-time stale-marker scan — a future regex change MUST NOT silently invalidate already-committed markers (which would re-fire HIGH on legitimately-deferred siblings with no deprecation window).
- **Boundary**: TB-005
- **Violated when**: the carve-out language is absent; the marker token is not pinned; the regex requires `:line-number` (it must be optional per the prose); the marker does not require per-sibling coverage; the worked example is absent; the comment-style list is presented as closed/exhaustive; `"""` is listed as a comment style; the syntactic-comment requirement is missing; the marker-format-migration clause is missing
- **Enforcement**: CI test assertion — the literal substring `SIBLING-DEFERRED:` appears in the section body; the regex with optional line-number group appears (literal `(:\d+)?` or equivalent); "per-sibling" or "each sibling" coverage language appears; a worked example appears inside a code fence within the section; the comment-style enumeration includes ≥6 styles including `<!-- -->` and `;` but excludes `"""`; "non-exhaustive" or "examples" prose appears within 5 lines of the comment-style enumeration; the prose names "true syntactic comment" or "start of a comment, not inside a string"; the prose names a deprecation-window or stale-marker-scan migration path
- **Guards against**: false positives blocking legitimate scope-limited fixes; AP-024 recurrence in the comment-style list (RS-015); RS-029 silent marker-format invalidation
- **Test approach**: unit
- **Risk**: medium — without the carve-out, the lens generates noise on every disciplined scope-limited fix, undermining its adoption; without a pinned marker token, the carve-out drifts into "reviewer judgment"
- **Note on the marker shape**: pinning `SIBLING-DEFERRED:` as the carve-out token does itself create a small enumeration risk. Mitigation: minimal syntax + non-exhaustive comment styles + CS-016 marker validity contract + the RS-029 migration clause. Future PRs amending the format must update both the agent prose and the test sub-assertion.
- **Implemented in**: (GREEN phase)

### CS-005: Calibrated HIGH severity for sibling-present findings (with worked-example contrast + aggressive default)

### CS-014: Per-fence size cap and truncation behavior (single emitted-bytes model, carved from the 100KB ceiling)
- **Type**: must
- **Category**: functional (resource-lifecycle)
- **Statement**: `/caudit` Step 6a emission of `<UNTRUSTED_FINDING_DESCRIPTION>` enforces caps measured on a **single canonical surface — the bytes actually emitted into the prompt**. There is one measurement model: count the bytes that will appear in the assembled prompt text (post-JSON-escape, including all object syntax, key strings, commas, outer brackets, and any truncation markers). Truncation iterates until the emitted bytes fit. No second measurement model is used at any layer.

  **Measurement primitive + locale (RS-022).** "UTF-8 byte length" is measured with a BYTE-counting primitive explicitly — `wc -c` or jq's `utf8bytelength`, NOT bash `${#var}` (which counts characters in UTF-8 locales) and NOT jq `length` (which counts codepoints). A multibyte fixture (a description of N multibyte characters whose UTF-8 encoding exceeds N bytes) is a required CS-013 test case to prove the byte (not char/codepoint) measurement. JSON assembly inherits the PMB-001 jq 1.7/1.8 precedence hazard and runs under the CI jq matrix; see EA-005.

  **Budget carve (RS-013).** The aggregate cap is carved from /caudit's 100 KB prompt ceiling AFTER the DIFF and RULES fences are measured — the fence is modeled as an ADDITION to a previously-passing budget, not an isolated 16 KB. The effective aggregate budget for the finding-description fence is `min(16384, 100KB − measured(DIFF) − measured(RULES) − overhead)`. A round that was previously under 100 KB MUST NOT abort solely because the 4th fence was added; if the carve would leave less than the fence's truncated-minimum, the fence is truncated to fit the carve (preserving a marker entry per finding) rather than aborting the round and blaming "prompt budget."

  **Per-entry cap**: the emitted JSON object for any single finding (`{"id":"<id>","description":"<escaped-description>"}`) MUST be ≤ 4096 emitted UTF-8 bytes. The truncation algorithm:

  1. Build the candidate emitted object (escape the description, format as JSON).
  2. Measure its UTF-8 byte length (byte-counting primitive) in the emitted form.
  3. If `≤ 4096`: keep as-is.
  4. If `> 4096`: drop one codepoint from the END of the description (do not split a multi-byte UTF-8 sequence, do not leave a partial JSON escape like `\u00`), append `[truncated: N more bytes]` (N = count of dropped raw-description bytes), re-escape, re-format, re-measure. Repeat. The loop terminates because each iteration strictly reduces description length; worst case the entry is `{"id":"<id>","description":"[truncated: NNNN more bytes]"}`.

  **Aggregate cap**: the emitted JSON array MUST be ≤ the carved budget (≤16384 bytes AND within the 100KB-ceiling carve). The algorithm:

  1. Build the emitted array with all per-entry-capped entries.
  2. Measure UTF-8 byte length of the emitted array text.
  3. If `≤ budget`: emit.
  4. If `> budget`: compute per-entry shares **proportionally** based on current emitted-byte length; re-truncate each entry's description to fit its share using the per-entry algorithm — **preserving at least a marker entry per finding (RS-013)** so the most-relevant (largest) class-shape description is NOT dropped first; rebuild; re-measure. Repeat at most 3 passes. Only if still over after 3 passes (adversarial escape-byte distribution) drop the smallest-emitted-bytes entry; retry. The fence is NEVER omitted just for being large — only when the array would be empty (per CS-011 empty-array omission).
  5. Emit the final array surrounded by fence delimiters.

  **The model in one sentence**: build the emitted text, measure its UTF-8 bytes with a byte-counting primitive, truncate proportionally (marker-per-finding preserved) until the measurement is at or under the carved budget.

  **Documentation**: the truncation algorithm is documented in `skills/caudit/SKILL.md` Step 6a prose with the literal numeric caps, the emitted-bytes model named explicitly, the byte-counting primitive named, and the 100KB-ceiling carve named.
- **Boundary**: AP-039 (unbounded data through bounded medium); TB-005 (fence assembly)
- **Violated when**: no per-entry cap; no aggregate cap; truncation marker missing; the measurement model is split; the measurement primitive is `${#var}`/jq `length` (char/codepoint, not byte); the truncation does not preserve UTF-8/JSON escape boundaries; the algorithm has no termination bound; the aggregate cap is modeled in isolation rather than carved from the 100KB ceiling after DIFF+RULES; truncation drops the largest finding before preserving a marker-per-finding; the size cap allows the prompt to push past /caudit's 100 KB hard ceiling
- **Enforcement**: CI test assertion — Step 6a prose contains the literal numeric caps (4096 / 16384 emitted bytes), the emitted-bytes model, the byte-counting primitive, and the 100KB carve; CS-013 fixtures verify per-entry truncation (≤4096), aggregate truncation (≤ carved budget), the escape-byte adversarial case (measurement on emitted form), the multibyte case (byte not codepoint), and a large-diff + max-fence case whose total is ≤100KB or whose abort attributes overflow to the fence

- **Test approach**: unit (Step 6a prose) + integration (CS-013 size-cap fixtures including escape-byte + multibyte adversarial cases)
- **Risk**: high — without a single consistent byte-measurement model carved from the real ceiling, the cap appears enforced but emitted bytes can silently exceed it or abort the round
- **Implemented in**: (GREEN phase)
