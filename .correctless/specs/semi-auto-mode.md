# Spec: Semi-Auto Mode

## Metadata
- **Task**: semi-auto-mode
- **Recommended-intensity**: high
- **Intensity**: high
- **Intensity reason**: touches hooks/ (file path signal), modifies shared constraints and state machine (workflow-advance.sh), orchestrates trust boundary between human judgment and LLM autonomy
- **Override**: none

## What

Semi-auto mode automates the implementation pipeline after human-approved spec review. The human owns "what" and "why" (spec, review); the LLM owns "how" (TDD, simplify, verify, arch-update, docs, PR). A new `/cauto` skill orchestrates the pipeline: `ctdd -> simplify -> cverify -> cupdate-arch -> cdocs -> PR creation`. A new `.correctless/preferences.md` file provides the LLM with codified judgment calls so it can make decisions the human would otherwise rubber-stamp. The system escalates to the human on architectural decisions, repeated failures, or spec contradictions.

This is the foundation layer for full-auto and factory modes.

## Rules

- **R-001** [integration]: `/cauto` must invoke skills in this exact order: `/ctdd`, `/simplify`, `/cverify`, `/cupdate-arch`, `/cdocs`. Each skill is invoked with a fresh context (`context: fork`) — not shared with the orchestrator or other skills. Each skill must complete successfully before the next begins. Before invoking `/simplify`, `/cauto` must create a commit ("TDD complete") to establish a clean revert point (see R-015). Verify by: (1) parsing the audit trail JSONL for `skill_started` and `skill_completed` entries in the expected order, (2) static grep confirming `context: fork` in each sub-skill's SKILL.md frontmatter.

- **R-002** [integration]: `/cauto` must only be invocable when the workflow phase is `review` (standard intensity) or `review-spec` (high+ intensity). Attempting to invoke `/cauto` in any other phase must produce an error message stating the required phase. This is intentional v1 scoping — future versions may support partial pipeline execution from later phases (e.g., `done` to run only verify+docs+PR, or `verified` to run only docs+PR). Verify by attempting to invoke `/cauto` in `spec`, `tdd-tests`, and `done` phases.

- **R-003** [integration]: `/cauto` must read `.correctless/preferences.md` before beginning the pipeline. If the file does not exist, `/cauto` must proceed with built-in defaults (not fail). Verify by: (1) static check that the SKILL.md contains a Read instruction targeting `.correctless/preferences.md`, (2) static check that the SKILL.md contains a fallback clause for when the file is missing, (3) integration test that `/cauto` does not abort when `preferences.md` is absent.

- **R-004** [unit]: `preferences.md` must support at minimum these preference categories: (a) QA finding triage — which severity levels to auto-fix vs surface, (b) documentation scope — what to include/exclude, (c) commit granularity — how to structure commits during implementation, (d) escalation sensitivity — what constitutes an architectural decision requiring human input, (e) PR creation — `gh` (default), `skip`, or a custom command. Verify by checking the template contains all five categories.

- **R-005** [integration]: When `/cauto` encounters a failure that persists after N attempts within a single skill, it must stop the pipeline, write a structured escalation summary to `.correctless/artifacts/escalation-{branch_slug}.md`, and wait for human input. The attempt threshold varies by effective intensity: standard=3, high=2, critical=2 (matching the calm reset thresholds from `/ctdd`). The escalation file uses YAML frontmatter for machine-parseable fields, with human-readable prose below. Required frontmatter fields: `completed_skills` (list), `failed_skill` (string), `failed_at_phase` (string), `failed_at_substep` (string, nullable — for inter-skill operations like post-simplify test run), `attempts_before_escalation` (integer), `pipeline_config` (object: intensity, preferences snapshot). Verify by simulating a persistent test failure at each intensity level and checking the escalation file is written with valid YAML frontmatter and the pipeline halts at the correct threshold.

- **R-006** [integration]: When `/cauto` detects a potential architectural decision during any skill, it must escalate to the human rather than making the decision autonomously. Detection heuristics: (a) the skill needs to add a new ABS-xxx or TB-xxx entry, (b) the gate hook blocks an ARCHITECTURE.md write outside the `cupdate-arch` phase (mechanical — treat any gate-blocked ARCHITECTURE.md write as an architectural escalation), (c) a spec rule cannot be satisfied without changing the spec, (d) a new dependency is required that isn't in the project, (e) the skill modifies CLAUDE.md (changes to the system's own instructions require human approval). Detection is best-effort for heuristics (a), (c), (d) — these are LLM judgment calls. The mechanical backstop is R-005: if an undetected architectural decision causes a skill to fail, the failure threshold catches it. Verify by: (1) attempting an ARCHITECTURE.md write outside `cupdate-arch` and confirming gate blocks it, (2) confirming `/cdocs` changes to CLAUDE.md trigger escalation.

- **R-007** [unit]: The shared constraint "Never auto-invoke the next skill" must remain the default behavior. `/cauto` overrides this constraint only for the skills within its pipeline and only when the user has explicitly invoked `/cauto`. The override mechanism: `/cauto` is the orchestrator that invokes skills — each skill still tells its caller "what comes next" and stops. The orchestrator, not the skill, decides to proceed. Skills invoked outside of `/cauto` must continue to respect the no-auto-invoke boundary. Verify by: (1) grep confirming `_shared/constraints.md` contains "Never auto-invoke", (2) grep confirming `/cauto` SKILL.md does not contain instructions to modify constraints.md, (3) running `/ctdd` standalone and confirming it ends with a recommendation, not auto-continuation.

- **R-008** [integration]: At the end of the pipeline, `/cauto` must create a PR according to the `pr_creation` preference in `preferences.md`. Options: `gh` (default) — create PR via `gh pr create`; `skip` — no PR, just report completion; custom command — run the specified command (TB-001b: custom PR commands are executed via shell, same trust model as TB-001a test runner commands). When creating a PR, the title must be derived from the spec's task name. The PR body must include: a summary section (auto-generated from spec and implementation), a test plan section (derived from spec rules), QA findings summary (rounds completed, findings fixed), and verification status. Verify by running the full pipeline with each PR creation mode.

- **R-009** [unit]: `/cauto` must respect the effective intensity computation from shared constraints. The intensity determines which skills in the pipeline are active: `cupdate-arch` only runs at high+ intensity (it has its own intensity gate in its SKILL.md). At standard intensity, the pipeline is: `ctdd -> simplify -> cverify -> cdocs -> PR`. Verify by running `/cauto` at standard intensity and confirming `cupdate-arch` is skipped, and by verifying `cupdate-arch/SKILL.md` contains an intensity gate independent of `/cauto`.

- **R-010** [integration]: `/cauto` must advance the workflow state machine through all required transitions using `workflow-advance.sh`. It must not skip transitions or write to the state file directly. The full transition sequence for a clean run using actual phase names: `tdd-tests -> tdd-impl -> tdd-qa -> (tdd-impl -> tdd-qa)* -> done -> verified -> documented`. At high+ intensity, the optional `tdd-verify` phase may occur between `tdd-qa` and `done` (handled internally by `/ctdd`). The `fix` command transitions back to `tdd-impl`, not a separate `fix` phase. Verify by checking the state file shows `documented` phase after a successful `/cauto` run and that the override log shows no overrides.

- **R-011** [integration]: `/cauto` must log its own orchestration decisions to the audit trail (`.correctless/artifacts/audit-trail-{branch_slug}.jsonl`). Each orchestration entry must have: `type` (one of: `skill_started`, `skill_completed`, `preference_applied`, `escalation_triggered`, `simplify_reverted`, `pipeline_completed`, `pipeline_failed`), `timestamp` (ISO 8601), `skill` (skill name or `orchestrator`), and `elapsed_ms` (milliseconds since pipeline start). Verify by parsing the JSONL after a run and validating each orchestration entry against this schema.

- **R-012** [integration]: `/simplify` (Claude Code built-in) runs between `/ctdd` and `/cverify`. `/simplify` operates outside the Correctless trust model — it has no SKILL.md, no allowed-tools, no `context: fork`, and no audit trail integration. It is treated as an untrusted contributor whose changes are validated post-hoc (R-015). Since `/simplify` has no workflow-advance.sh transition, `/cauto` invokes it after `/ctdd` completes (phase `done`) and before invoking `/cverify`. The gate hook allows all edits during `done` phase. Verify by confirming the workflow state is `done` both before and after `/simplify`.

- **R-013** [unit]: `/csetup` must scaffold a default `preferences.md` during project initialization. The scaffold must be a commented template showing all preference categories with sensible defaults. If `preferences.md` already exists, `/csetup` must not overwrite it (idempotency). Verify by running `/csetup` twice — first creates the file, second skips it.

- **R-014** [advisory]: `/cauto` must emit progress updates between each skill invocation. Progress is observable via R-011 audit trail entries which include `elapsed_ms` and skill transition data. Conversational progress updates (printed to the user) are best-effort LLM behavior and not mechanically testable.

- **R-015** [integration]: Before invoking `/simplify`, `/cauto` must commit all TDD changes ("TDD complete") to establish a clean revert point. After `/simplify` completes, `/cauto` must: (1) check the diff for modifications to `.correctless/` paths and config files — if found, revert with `git reset --hard HEAD` and continue without simplification, (2) re-run the project's test suite — if tests fail, revert with `git reset --hard HEAD` and continue without simplification. If tests pass and no protected paths were modified, accept the simplification. Log all revert decisions in the audit trail (R-011, type `simplify_reverted`) but do not escalate. Verify by: (1) creating a test-breaking change after the TDD commit and confirming `git reset --hard HEAD` restores the TDD state, (2) confirming modifications to `.correctless/` paths trigger revert.

- **R-016** [integration]: `/cauto` must support resumption. On invocation, `/cauto` checks for an existing escalation file at `.correctless/artifacts/escalation-{branch_slug}.md`. If present, it parses the YAML frontmatter and checks: (a) `failed_at_phase` matches the current workflow phase, (b) completed skill artifacts still exist (tests pass, verification report present if applicable). If both checks pass, `/cauto` resumes from the failed skill. If either check fails (phase inconsistent or artifacts missing), `/cauto` deletes the stale escalation file and starts fresh. Verify by: (1) triggering an escalation in `/cverify`, resolving the issue, re-invoking `/cauto`, and confirming it resumes at `/cverify` without re-running `/ctdd`; (2) triggering an escalation, manually resetting the workflow, re-invoking `/cauto`, and confirming it starts fresh.

- **R-017** [integration]: When `/ctdd` triggers a spec-update (the spec was found to be wrong during TDD), `/cauto` must escalate to the human. The escalation summary must state which spec rule is unsatisfiable and why. This resets the phase to `spec`, which is human-gated territory. After the human fixes the spec and re-approves review, they re-invoke `/cauto` — resumption (R-016) detects the phase is `review`/`review-spec` and starts the pipeline fresh (no stale escalation, clean state). Verify by simulating a spec-update trigger during TDD and confirming the escalation contains the problematic rule and the pipeline halts.

- **R-018** [integration]: At pipeline startup, `/cauto` must verify tool availability for configured operations. If `pr_creation` preference is `gh` (or default), check `command -v gh` and fail fast with: "gh CLI not installed. Set `pr_creation: skip` in preferences.md or install gh." This check runs before any skill invocation to prevent hours of implementation followed by a PR creation failure. Verify by running `/cauto` with `pr_creation: gh` when `gh` is not on PATH and confirming immediate failure with actionable message.

- **R-019** [unit]: `.correctless/preferences.md` must be added to the sensitive-file-guard's protected file list. The file contains commands that get executed (custom PR creation commands per R-008, TB-001b) and preference directives that control autonomous agent behavior. It requires the same protection as `workflow-config.json`. Verify by confirming the sensitive-file-guard blocks Write/Edit to `preferences.md` during gated phases.

## Won't Do

- **Full-auto mode** — no agent health monitoring (paralysis/churn detection), no notification hooks, no auto-branch creation. Those are full-auto features that build on this foundation.
- **Factory mode** — no spec-from-intent generation, no CI integration, no auto-merge. Future tier.
- **Preference learning** — preferences are static in `preferences.md`. No automatic preference inference from human behavior. Future enhancement.
- **Parallel skill execution** — skills run sequentially. No concurrent TDD + verify. The state machine is linear.
- **Custom pipeline ordering** — the skill sequence is fixed. No user-configurable pipeline. The order was chosen for correctness (verify after simplify, docs after verify).
- **Partial pipeline from later phases** — v1 only starts from `review`/`review-spec`. Starting from `done` or `verified` to run a subset of the pipeline is a future enhancement (see R-002 note).
- **Escalation notifications** — v1 blocks and waits for the human. Factory mode will need timeout-based notifications (desktop/webhook) for when the human is unavailable. The escalation file format (R-005) is designed to support this future need — it contains all the context a notification system would need to compose a message.

## Risks

- **Context window exhaustion** — at high+ intensity, the orchestrator is expected to require 1-2 resumptions due to context limits. Each skill runs in forked context (R-001), so context accumulates only in the orchestrator, but skill results, audit trail entries, and state management still consume orchestrator context. The orchestrator monitors its own context per the shared constraint (warn at 70%, stop at 85%). If context is exhausted mid-pipeline, it triggers R-005 escalation with pipeline state so the human can resume via R-016. This is acceptable for semi-auto — factory mode will address this via per-session container management.
- **Preference drift** — preferences.md may go stale as the project evolves. Mitigation: accepted for semi-auto. Preference learning is a full-auto concern.
- **Skill failure cascading** — a failure in an early skill (ctdd) blocks the entire pipeline. Mitigation: this is by design. The pipeline is sequential because correctness requires it. The escalation mechanism (R-005) and resumption (R-016) handle failures without wasting completed work.
- **`/simplify` outside trust model** — `/simplify` is a Claude Code built-in operating outside Correctless conventions. It has no SKILL.md, no allowed-tools, no `context: fork`. Mitigation: treated as an untrusted contributor — changes are committed before `/simplify` runs, post-simplify diff rejects modifications to `.correctless/` and config paths, test re-run catches functional breakage, `git reset --hard HEAD` provides atomic revert (R-015).

## Design Notes

- **Escalation file as the resumption checkpoint**: The escalation file (R-005) serves double duty — it's both the human-readable summary and the machine-readable checkpoint for resumption (R-016). YAML frontmatter contains structured fields (`completed_skills`, `failed_skill`, `failed_at_phase`, `failed_at_substep`, `attempts_before_escalation`, `pipeline_config`), prose section below contains the human-readable explanation. This avoids a separate checkpoint format while supporting both audiences.
- **Preference override via CLAUDE.md**: Project preferences live in `.correctless/preferences.md`. A contributor who wants personal overrides can put them in their user-level CLAUDE.md. The LLM naturally applies CLAUDE.md directives, so personal preferences override project defaults without any special mechanism.
- **Factory mode readiness**: The escalation file format, preference surface, and resumption protocol are all designed to support factory mode. Factory adds: escalation notifications (timeout + webhook), CI integration (run tests in CI not locally), and auto-merge (after human approves PR). None of these require architectural changes to semi-auto — they layer on top.
- **`/simplify` trust model**: Built-in Claude Code commands invoked by the orchestrator operate outside the Correctless trust model. Their effects are validated post-hoc (R-015), not pre-gated. This is analogous to reviewing an untrusted contributor's PR — accept if clean, revert if not.
- **TB-001b: Custom PR commands**: The `pr_creation` custom command in `preferences.md` is structurally identical to TB-001a (test runner commands from config). Same trust model: the file is local, under the project owner's control, and protected by the sensitive-file-guard (R-019). If the trust model extends to shared or marketplace preferences, this exception must be revisited.

## Prerequisites

- **Fix `is_full_mode()` to consult effective intensity**: Pre-existing state machine bug — `is_full_mode()` checks `workflow.intensity` in config only, ignoring `feature_intensity` from the state file. This means per-feature intensity elevation (dynamic rigor PAT-005) is broken for phase routing. Fix as a separate prerequisite, not part of this spec. Affects all features at mixed intensity, not just `/cauto`.
- **Add `context: fork` to `cdocs` and `cupdate-arch` frontmatter**: Both skills lack `context: fork`. R-001 requires fresh context for every pipeline skill. Add during implementation.
- **Add PAT-007 to ARCHITECTURE.md**: Conditional update path testing (guards AP-002). Approved during spec review.
- **Add PAT-008 to ARCHITECTURE.md**: Idempotent migration testing (guards AP-004). Approved during spec review.
- **Add TB-004 to ARCHITECTURE.md**: LLM orchestrator autonomy boundary — documents the trust boundary between human spec approval and LLM autonomous execution. Invariant: the LLM must escalate when it encounters decisions outside its delegated authority (R-006 heuristics). Violated when the LLM silently makes an architectural decision without escalating.
- **Add ABS-007 to ARCHITECTURE.md**: Escalation file contract — `/cauto` is sole writer, schema defined in R-005, consumers are `/cauto` (resumption) and the human (context).
- **Add ABS-008 to ARCHITECTURE.md**: preferences.md contract — `/csetup` scaffolds, `/cauto` and all pipeline skills read, human edits. Fallback to defaults when missing.
- **Add ENV-004 to ARCHITECTURE.md**: `gh` CLI as optional dependency — required when `pr_creation: gh` (default). Consequence if missing: pipeline completes but PR creation fails. R-018 mitigates with upfront check.
- **Add preference reading to shared constraints**: "If `.correctless/preferences.md` exists, read it and apply project preferences." This gives every skill access to preferences without a propagation mechanism.
