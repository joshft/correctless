The standard workflow enforces a linear pipeline: spec → review → TDD → verify → docs → merge. Each step is a separate skill invocation. The human decides when to advance between steps — skills never auto-invoke the next skill.

At high+ intensity, the pipeline expands: /creview becomes /creview-spec (4-agent adversarial review), /cupdate-arch runs after verification, and /caudit (Olympics convergence audit) runs before merge.

The state machine in hooks/workflow-advance.sh enforces the ordering — you cannot skip phases or go backwards. Each transition has a gate that validates preconditions (e.g., tests must fail before advancing from RED to GREEN, verification report must exist before advancing to documented).

Five hooks run on every tool call, in this order:

  1. hooks/sensitive-file-guard.sh (401 lines) — PreToolUse. Blocks writes to .env, credentials, keys, certificates. Fail-closed, no overrides. Runs before workflow-gate so sensitive files are always protected regardless of workflow phase. Hardcoded 20 default patterns + custom patterns from config.
  2. hooks/workflow-gate.sh (550 lines) — PreToolUse. Enforces phase-specific file restrictions. During RED, only test files and STUB:TDD source stubs are allowed. During QA, all edits are blocked. During spec/review, source and test files are locked.
  3. Tool executes (only if both PreToolUse hooks exit 0)
  4. hooks/audit-trail.sh (208 lines) — PostToolUse. Logs every file modification with phase context to a JSONL trail file. Tracks adherence metrics for QA coverage analysis. 5MB rotation cap.
  5. hooks/auto-format.sh (149 lines, at .claude/hooks/) — PostToolUse. Runs the project's formatter (Prettier, Black, gofmt, etc.) on the edited file. Exact-match allowlist validation, array-based execution.

hooks/statusline.sh (265 lines) renders continuously, showing: branch, phase, QA round, cost, context %, lines delta.

All hooks follow PAT-001: set -euo pipefail + set -f, jq availability check, bulk eval+jq @sh stdin parse, fast-path exit 0 for non-relevant tools before loading config.

The TDD cycle is managed by /ctdd, which is an orchestrator — it spawns separate agents for each phase and never writes code itself. This enforces agent separation: the test agent (RED) never sees the implementation plan, the implementation agent (GREEN) never sees test-writing context, and the QA agent is independent of both.

RED phase: The test agent writes failing tests encoding every spec rule (INV-xxx, PRH-xxx, BND-xxx). The workflow gate (workflow-gate.sh:469) blocks source file edits unless they contain STUB:TDD markers. After tests exist, a test audit agent checks test quality before implementation begins — this catches mock gaps, weak assertions, and missing integration tests.

GREEN phase: The implementation agent makes the failing tests pass. It can edit source and test files (logged). The gate validates with commands.test_new at the qa transition (cmd_qa:483). A calm reset prompt fires after 3 consecutive failures to redirect the agent away from dead-end approaches.

QA phase: An independent QA agent reviews the implementation with a hostile lens. All edits are blocked (workflow-gate.sh:525). Every BLOCKING finding must include both an instance fix (fix this bug) and a class fix (prevent this category). The fix → re-QA loop runs up to 3 rounds at high intensity. Findings are persisted to .correctless/artifacts/qa-findings-{slug}.json.

The gating decision tree has multiple early exits for performance. The most common path (non-write tools like Read/Grep) exits at the fast-path check before loading any config or state. For write tools, the hook reads the state file once (bulk jq), classifies the target file (test/source/other via config patterns), then applies phase-specific rules.

Phase-specific rules (workflow-gate.sh:458):

  • spec/review/model: All source and test edits blocked
  • tdd-tests (RED): Source blocked unless contains STUB:TDD; test files allowed
  • tdd-impl (GREEN): Source and test files allowed (test edits logged)
  • tdd-qa/tdd-verify: All edits blocked (QA review in progress)
  • audit: Source and test files allowed (fixing audit findings)

The override mechanism (workflow-advance.sh override "reason") grants 10 tool calls that bypass gating. Override usage is logged and the count is tracked — 2+ overrides triggers a warning in /cstatus.

All workflow artifacts are branch-scoped (the branch slug + hash is embedded in filenames). This allows concurrent workflows on different branches without interference. The workflow-advance.sh reset command cleans up all branch-scoped artifacts.

Key artifact flows:

  • Spec → State: cmd_init creates the state file referencing the spec path
  • State → Gate: workflow-gate.sh reads the state file to determine the current phase
  • Gate → Audit trail: audit-trail.sh logs every file modification with phase context
  • QA findings → Verification: /cverify reads QA findings to verify class fixes were implemented
  • Config → Everything: workflow-config.json provides test commands, file patterns, intensity level, and hook settings
